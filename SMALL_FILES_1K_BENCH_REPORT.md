# 1000 小文件性能测试报告（集中存储）

**生成时间：** 2026-07-01T08:30:00Z
**测试对象：** fssvrgo 分布式文件存储服务
**测试范围：** 集中存储（LocalStorage）· 写入 1000 个 1KB 文件后回读 · HTTP 与 gRPC 双协议覆盖
**代码版本：** 含 gRPC 小文件上传快速路径优化（commit `9c38405`）

> 结论速览：4 个阶段共 **4,000 次操作全部成功，0 错误**。
> - **写入：gRPC 比 HTTP 快 1.96×**（2455 vs 1254 files/s）——gRPC 快速路径绕过会话机制后反超 HTTP；
> - **读取：gRPC 比 HTTP 快 1.08×**（5293 vs 4899 files/s）——两者读路径相同，差异在传输层；
> - **总耗时仅 1.81 秒**，集中存储在小规模场景下延迟极低。

---

## 1. 测试目标

按要求对当前项目进行性能压测：

- 写入 **1000 个 1KB 大小**的文件，随后读回这些文件；
- **只测试集中存储**（本地文件系统 LocalStorage）；
- **使用 Redis 作为一致性锁**（分布式 `SET NX` 加锁 + Lua 原子解锁）；
- **使用 PostgreSQL 作为元数据数据库**；
- **HTTP 与 RPC（gRPC over TCP）两种方式都要覆盖**——必须是真实的 gRPC TCP 调用；
- 出具本测试报告。

## 5. 对比分析

### 5.1 HTTP vs gRPC

| 指标 | HTTP | gRPC | 倍率 |
|------|------|------|------|
| 写吞吐 (files/s) | 1253.8 | 2455.3 | **gRPC 快 1.96×** |
| 读吞吐 (files/s) | 4898.7 | 5293.1 | gRPC 快 1.08× |
| 写 p50 (μs) | 7461 | 4307 | **gRPC 低 42%** |
| 写 p95 (μs) | 11767 | 6285 | **gRPC 低 47%** |
| 写 p99 (μs) | 95792 | 39537 | **gRPC 低 59%** |
| 读 p50 (μs) | 2062 | 1963 | gRPC 低 5% |
| 读 p95 (μs) | 4618 | 4207 | gRPC 低 9% |
| 读 p99 (μs) | 10731 | 10489 | gRPC 低 2% |

**分析：**

- **写入：gRPC 全面胜出（吞吐 1.96×，p99 延迟低 59%）。** 与 50 万文件测试结论一致——gRPC 快速路径对单分块完整上传直接调用 `FileManager.UploadFile`，绕过 transferSvc 会话机制，HTTP/2 长连接 + 多路复用使单次操作基准成本更低。
- **读取：两者接近持平（gRPC 快 1.08×）。** 1000 文件规模下读延迟以 PG 查询 + `os.ReadFile` 为主，传输层差异被淡化。gRPC 凭 HTTP/2 长连接略占优。
- **小规模特征：** 1000 文件总耗时仅 1.81 秒，冷启动效应（PG 连接池预热、Redis 首次连接、OS 页缓存冷）对结果有一定影响，尤其是 HTTP Write 的 p99=95792μs 明显高于大规模测试的 p99=44142μs，反映出小样本下尾部延迟波动更大。

### 5.2 与 50 万文件测试对比

| 指标 | 1000 文件 (files/s) | 50 万文件 (files/s) | 比值 |
|------|---------------------|---------------------|------|
| HTTP Write | 1253.8 | 1095.1 | 1.14× |
| HTTP Read | 4898.7 | 5520.2 | 0.89× |
| gRPC Write | 2455.3 | 2352.7 | 1.04× |
| gRPC Read | 5293.1 | 4572.3 | 1.16× |

**分析：** 1000 文件与 50 万文件的吞吐量在同一量级，说明系统的性能特征在不同规模下保持稳定。写吞吐在小规模略高（无大规模并发争用），读吞吐在小规模略低（冷页缓存效应）。gRPC Write 在两种规模下均稳定在 2350~2455 files/s，证明快速路径优化效果可靠。

## 6. 关键发现

1. **零错误：** 4,000 次操作全部成功，PostgreSQL + Redis 锁 + 本地存储组合在小规模场景下完全可靠。
2. **gRPC 写入优势在小规模依然成立：** 快速路径使 gRPC 写吞吐达 HTTP 的 1.96×，与 50 万文件测试的 2.15× 一致。
3. **读取性能两者持平：** 1000 文件规模下读路径开销以 DB + 磁盘为主，协议差异被淡化。
4. **小样本尾部延迟波动大：** HTTP Write p99=95792μs 高于大规模测试，受冷启动和 GC 暂停影响，属正常现象。
5. **总耗时 1.81 秒：** 集中存储在小规模场景下性能优异，适合作为与 MinIO 对象存储对比的基线。
